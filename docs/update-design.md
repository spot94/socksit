# SocksIt — обновления и канал поставки (дизайн)

Статус: **черновик на согласование** (кода ещё нет). Ниже — формат манифеста, протокол
проверки/применения, модель угроз, изменения конфига/UI и шаги CI. После согласования —
поэтапная реализация (см. §11).

Требования-«пол» из [CLAUDE.md](../CLAUDE.md), которые здесь важны: код и сборка в **GitLab CI/CD**
(CICD-1/2), секреты только в CI/Vault (**SEC-1**), граница периметра и запрет «звонить» наружу без
основания (**DATA-2**), аудит действий (**SEC-3**). Обновление = вектор исполнения кода, поэтому
подлинность важнее способа доставки.

---

## 1. Цели и объём

- Безопасно обновлять `socksit.exe` (единый бинарь: служба + панель + трей) и, реже, движок
  `sing-box.exe`.
- Источник обновлений — **корпоративный on-prem endpoint**, публикуемый GitLab CI. Никакого
  публичного интернета и телеметрии по умолчанию.
- Подлинность гарантируется **криптоподписью манифеста**, а не доверием к серверу/каналу.

**Не в объёме:** обновление конфигов/пресетов (они пользовательские, не трогаются), авто-даунгрейд,
дельта-патчи (обновляем целым файлом), сторонние зеркала.

---

## 2. Что обновляется

| Компонент | Как заменяется | Частота |
|---|---|---|
| `socksit.exe` | self-replace службы (см. §7) | по релизам |
| `sing-box.exe` (+`libcronet.dll`, если используется) | служба останавливает дочерний процесс, меняет файл, запускает | редко (смена pinned-версии) |
| Конфиг/пресет | **не трогаются** | — |

Движок и приложение версионируются независимо (манифест несёт обе версии).

## 3. Канал поставки

**Выбран профиль: GitHub Releases + GitHub Actions** (личное/открытое распространение).

```
GitHub repo ──Actions──▶ build ▶ sign exe (signtool) ▶ sha256
                              ▶ build manifest.json ▶ sign Ed25519 (manifest.json.sig)
                              ▶ Release с ассетами: socksit.exe, sing-box.exe, manifest.json(.sig), MSI
                                              │
  клиент (служба) ◀── HTTPS GET (напрямую / через прокси, §9.1) ──┘
                       <owner>/<repo>/releases/latest/download/manifest.json(.sig)
```

- **Endpoint & транспорт** — базовый URL, откуда доступны `manifest.json`, `.sig` и артефакты. Для
  GitHub это `https://github.com/<owner>/<repo>/releases/latest/download` (всегда последний релиз,
  API не нужен). GitHub раздаёт по HTTPS; целостность и подлинность всё равно держит подпись (§5), а
  не TLS — GitHub для нас недоверенный CDN.
- **Каналы**: `stable` (по умолчанию), опц. `beta` — просто разные пути/манифесты.
- **MSI** ([installer.wxs](../build/installer.wxs), `MajorUpgrade`) остаётся для первичной установки
  и централизованного пуша (SCCM/Intune/GPO). Для таких парков встроенный апдейтер можно выключить
  (`update.mode: off`) и катить через MSI.

### 3.1. Канал сменный (endpoint-агностичный)

Доверие держится на **подписи манифеста (§5)**, а не на хосте, поэтому источник — просто базовый URL
в `update.endpoint`, откуда доступны `manifest.json`, `manifest.json.sig` и артефакты. Профили:

| Профиль | `endpoint` (пример) | CI | Когда |
|---|---|---|---|
| **On-prem** (реком. для контура) | `https://updates.corp.local/socksit/stable` | GitLab CI | закрытый корпоративный периметр |
| **GitLab Pages/Registry** | `https://<gitlab>/socksit/-/releases/…` | GitLab CI | внутри, но без своего сервера раздачи |
| **GitHub Releases** | `https://github.com/<owner>/<repo>/releases/latest/download` | GitHub Actions | личное/открытое/вне закрытого контура |

**GitHub Releases** удобен тем, что URL `…/releases/latest/download/<asset>` всегда указывает на
последний релиз — клиенту не нужен API GitHub и отдельный код: те же `manifest.json` + `.sig` + `.exe`
кладутся ассетами релиза, `endpoint` = эта база. Подпись Ed25519 + sha256 работают без изменений
(GitHub — недоверенный CDN). Каналы `stable`/`beta` — через разные репозитории или имена ассетов
(`manifest-beta.json`).

> **Ограничение по [CLAUDE.md](../CLAUDE.md):** GitHub — внешний публичный сервис. Это расходится с
> **CICD-1** («код в GitLab») и **DATA-2** (граница периметра: машины ходят за обновлением в публичный
> интернет; в закрытых контурах часто заблокировано, артефакты/код покидают контур). Использовать
> GitHub для закрытого контура допустимо **только записанным решением человека** (исключение в
> манифесте) — самовольно требование не понижаем. Для личного/открытого распространения GitHub —
> подходящий выбор по умолчанию.

## 4. Формат манифеста

`GET <endpoint>/<channel>/manifest.json` — обычный JSON:

```json
{
  "schema": 1,
  "product": "socksit",
  "channel": "stable",
  "version": "0.2.0",
  "released": "2026-07-20T10:00:00Z",
  "min_supported": "0.1.0",
  "notes_en": "Fixes X; adds Y.",
  "notes_ru": "Исправлено X; добавлено Y.",
  "app": {
    "url": "https://updates.corp.local/socksit/stable/socksit-0.2.0-amd64.exe",
    "sha256": "9f2b…",
    "size": 11761152
  },
  "engine": {
    "version": "1.13.14",
    "url": "https://updates.corp.local/socksit/engine/sing-box-1.13.14-windows-amd64.exe",
    "sha256": "a1c4…"
  }
}
```

Подпись — **отдельным файлом** `GET <endpoint>/<channel>/manifest.json.sig`:
base64(Ed25519-подпись над **точными байтами** `manifest.json`). Отдельная подпись, а не поле внутри,
— чтобы не зависеть от канонизации JSON.

Поля:
- `version` — semver `MAJOR.MINOR.PATCH`, монотонно растёт.
- `min_supported` — клиенты старше не могут доехать самообновлением; им нужен реинсталл через MSI
  (сервер отдаёт понятное сообщение).
- `app`/`engine.url` — абсолютные URL внутри периметра; `sha256` — обязателен.
- `notes_*` — заметки на двух языках (панель показывает по выбранному языку).

## 5. Подпись и проверка

- **Ключи:** пара **Ed25519**. Приватный ключ — только в секретах CI (**GitHub Actions Secret /
  Environment**, для on-prem — GitLab CI/Vault; SEC-1), нигде в репо и образах. Публичный ключ **вшит
  в бинарь** (константа; сборочный override через ldflags для разных контуров). Ключ в конфиге НЕ
  хранится — иначе правкой YAML можно подменить доверие.
- **Ротация:** в бинарь вшиваем **список** доверенных публичных ключей (текущий + следующий). Смена
  ключа = релиз с добавленным новым ключом → позже вывод старого. Так ротация не ломает клиентов.
- **Проверка на клиенте (строгий порядок, любой сбой = отказ):**
  1. скачать `manifest.json` + `.sig`;
  2. проверить Ed25519-подпись над байтами манифеста хотя бы одним вшитым ключом;
  3. `schema` поддержан, `product == "socksit"`, `channel` совпадает;
  4. `version > current` (иначе — нечего делать; см. анти-даунгрейд §6);
  5. `current >= min_supported` (иначе — «обновитесь через установщик»);
  6. скачать артефакт, проверить `sha256`;
  7. (опц., прод) проверить **Authenticode**-подпись exe и издателя;
  8. только теперь применять (§7).
- Подпись самих бинарей (Authenticode) — обязательна в проде: снимает предупреждения SmartScreen и
  срабатывания AV (наш прошлый инцидент PDM:Trojan).

---

## 6. Модель угроз

| Угроза | Митигируется |
|---|---|
| Подмена/компрометация сервера обновлений | Ed25519-подпись манифеста: без приватного ключа валидный манифест не подделать |
| MITM на HTTP | То же (подпись); HTTPS — второй рубеж |
| Подмена артефакта | `sha256` из подписанного манифеста (+ Authenticode) |
| Вредоносный/подменный прокси (при `update.proxy`) | Прокси — недоверенный транспорт; подпись §5 сквозная. Может только блокировать/отдать устаревшее, не подделать |
| Даунгрейд/replay старого **валидно подписанного** манифеста | Клиент хранит max увиденную версию и отказывается ставить ≤; `min_supported`; (опц.) проверка свежести по `released`. **Остаточный риск:** Ed25519 сам по себе не защищает от повторной подачи старого подписанного манифеста — закрываем «храним-максимум-версии». |
| Правка YAML для подмены источника/ключа | Публичный ключ вшит (не в конфиге); endpoint из конфига влияет только на то, *откуда* тянуть, но не на *доверие* |
| Компрометация приватного ключа | Ротация (список ключей), инцидент-процедура: выпустить релиз с новым ключом, вывести старый |
| Утечка данных наружу (DATA-2) | endpoint по умолчанию пуст → апдейтер выключен; включается только явной настройкой на on-prem URL; никакой телеметрии |
| Порча при применении | Бэкап `.old` + health-check + автооткат (§7) |

## 7. Применение (self-update службы)

Служба под LocalSystem не может заменить свой запущенный exe напрямую, но имеет право писать в
`Program Files`. Паттерн **rename-replace-restart** без отдельного elevated-хелпера:

1. скачать новый `socksit.exe` в `%ProgramData%\SocksIt\update\` (staging), проверить (§5);
2. `MoveFile` текущего `Program Files\SocksIt\socksit.exe` → `socksit.exe.old` (переименование
   запущенного exe в Windows разрешено);
3. скопировать новый на его место (+ движок, если обновляется);
4. записать в аудит-лог (SEC-3: кто/что/над каким объектом → «обновление до vX»);
5. запросить у SCM **перезапуск** службы → стартует новый бинарь; трей-keeper поднимет трей заново;
6. **health-check:** новый процесс за N сек должен подняться и сгенерировать валидный `config.json`;
7. при успехе — удалить `.old` (при следующем старте); при неудаче — **откат**: вернуть `.old` и
   перезапуститься.

Движок: остановить дочерний процесс → заменить файл → запустить (служба им и так управляет).

**Влияние на трафик:** перезапуск службы на секунды роняет туннель. Если включён kill-switch,
проксируемые приложения на это время заблокированы (ожидаемо). Поэтому режим `auto` применяет
обновления в «тихое окно»/по расписанию, а `notify` — только по кнопке в панели.

## 8. Изменения конфига

Новый блок в `socksit.yaml` (парсится [internal/config](../internal/config/schema.go)):

```yaml
update:
  endpoint: ""            # базовый URL релизов; пусто = обновления ВЫКЛ (безопасно из коробки, DATA-2)
                          # GitHub: https://github.com/<owner>/<repo>/releases/latest/download
  channel: "stable"       # stable | beta
  mode: "notify"          # off | notify | auto
  check_interval: "24h"   # период автопроверки
  proxy: ""               # как ходить на endpoint (см. §9.1):
                          #   ""         — напрямую
                          #   "system"   — системный прокси (env HTTP(S)_PROXY; позже — настройки IE/WinHTTP)
                          #   "use-socks"— через настроенный SOCKS5 (proxy.address:port + креды из DPAPI)
                          #   "socks5://host:port" | "http://host:port" — явный прокси
```

- Дефолт `endpoint: ""` → апдейтер выключен, ничего наружу не ходит (безопасно из коробки).
- В **пресете** сборки endpoint проставляется на нужный GitHub-репозиторий; при необходимости —
  `proxy: use-socks`, если до GitHub можно достучаться только через сам SOCKS5-прокси.
- Публичный ключ(и) — **не здесь**, а вшиты в бинарь.

## 9. Поведение клиента

- Проверку ведёт **служба** (у неё есть сеть и права применить); панель/трей только показывают статус
  и шлют «проверить сейчас»/«применить» через IPC.
- Расписание: раз в `check_interval` + по требованию из панели. Джиттер, чтобы парк не бил сервер
  одновременно.
- `mode`:
  - `off` — не проверять (обновления через MSI/пуш);
  - `notify` — проверять и показывать «доступна vX», применять только по кнопке;
  - `auto` — скачать, проверить, применить по расписанию (с учётом §7).
- Все проверки — best-effort и молча-логируемые; сбой сети не мешает основной работе.

### 9.1. Обращение к endpoint через прокси

Проверку/скачивание ведёт **служба**; она собирает `*http.Client` с транспортом по `update.proxy`:

| `update.proxy` | Транспорт |
|---|---|
| `""` | напрямую (дефолтный транспорт) |
| `"system"` | системный прокси — на старте `http.ProxyFromEnvironment` (env `HTTP(S)_PROXY`/`NO_PROXY`); настройки IE/WinHTTP — позже |
| `"http://host:port"` | `Transport.Proxy = http.ProxyURL(u)` |
| `"socks5://host:port"` | SOCKS5-диалер (`golang.org/x/net/proxy`) в `Transport.DialContext` |
| `"use-socks"` | тот же SOCKS5-диалер, но параметры берутся из `proxy.address:port` (+ логин/пароль из DPAPI) |

Важное: служба обращается к SOCKS5-серверу **напрямую по TCP** (её собственный трафик исключён из TUN
anti-loop-правилом), поэтому `use-socks` работает даже когда туннель не поднят или проксирование на
паузе. Сценарий `use-socks` — для сетей, где до GitHub можно достучаться только через сам SOCKS5.

Новая зависимость: `golang.org/x/net` (SOCKS5-диалер). Прокси — это лишь недоверенный транспорт;
сквозную подлинность держит подпись (§5): вредоносный/подменный прокси может только заблокировать или
отдать устаревшее, но не подделать обновление.

## 10. UX

Новый раздел **Updates** в панели (или в About): текущая версия · версия движка · «Проверить сейчас»
· при наличии — «Доступна vX» + заметки (по языку) + «Применить» · настройки `mode`/`check_interval`
(endpoint/channel — обычно из пресета, но можно показать readonly). Прогресс и результат — строкой,
без отдельных окон (в духе единого окна).

Новые IPC-операции: `OpUpdateStatus` (последняя проверка, доступная версия), `OpUpdateCheck`,
`OpUpdateApply`.

## 11. Этапы реализации

1. ✅ **Проверка + notify (реализовано):** конфиг `update.*` ([config](../internal/config/schema.go)),
   клиент проверки подписанного манифеста ([internal/updates](../internal/updates/updates.go), Ed25519 +
   версии, покрыт тестами), выбор транспорта по `update.proxy` (`""`/`system`/`use-socks`/явный), периодическая
   проверка в службе (notify), IPC `update_status`/`update_check`, блок **Updates в About**. Публичный ключ
   вшивается через ldflags (`-X socksit/internal/updates.PublicKeyB64=<b64>`) — пока пустой, до Этапа 3.
   Применение (скачивание/self-replace) — **не** в этом этапе.
2. **Применение + откат:** self-replace службы (§7), health-check, автооткат, аудит.
3. **CI-пайплайн (GitHub Actions):** по тегу `vX.Y.Z` — сборка с версией через ldflags (`-X main.Version`),
   подпись exe (signtool), sha256, сборка+подпись манифеста (Ed25519, приватный ключ из Actions Secret),
   создание GitHub Release с ассетами (`socksit.exe`, `sing-box.exe`, `manifest.json`, `.sig`, MSI).
   `endpoint` клиентов указывает на `…/releases/latest/download`.
4. **Движок:** обновление `sing-box.exe` по манифесту.
5. **Каналы/ротация ключей**, `beta`.

## 12. Открытые вопросы (на решение)

- ✅ **Профиль канала:** GitHub Releases (личное/открытое распространение). Если позже понадобится
  закрытый контур — включить on-prem-профиль и зафиксировать исключение по CICD-1/DATA-2 в манифесте.
- ✅ **CI-платформа:** GitHub Actions.
- **Репозиторий** `<owner>/<repo>` для релизов (нужен, чтобы прописать дефолтный `endpoint`).
- **Приватный Ed25519-ключ** в GitHub Actions Secret напрямую или через окружение с ревью
  (Environment protection rules)?
- **`update.proxy` по умолчанию:** `""` (напрямую) или `system`? (`use-socks` — по ситуации в пресете.)
- **Authenticode** сейчас (нужен сертификат подписи кода — [signing.md](signing.md)) или позже?
- **Updates** в UI — отдельный пункт сайдбара или блок внутри **About**?
- Endpoint: HTTP или HTTPS в проде; нужен ли auth (mTLS/токен) к серверу обновлений?
- `mode` по умолчанию: `notify` (рекомендую) или `off` до явного включения?
- Нужен ли Authenticode прямо сейчас (нужен сертификат подписи кода — см. [signing.md](signing.md))?
- Раздел **Updates** — отдельный пункт сайдбара или блок внутри **About**?
