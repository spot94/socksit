# SocksIt

## Как читать этот файл

Ниже — **неснижаемый минимум требований** компании к ИИ-проектам. Это пол, а не полный план.
Соблюдай его с первого коммита. Детальные требования под конкретный проект и разбивку по
задачам добавит скилл ce-brainstorm — он же разрешит условные пункты (веб/LDAP/периметр и т.п.).

**YOU MUST** держаться этих правил. Требование можно снять как N/A только записанным решением
человека в манифесте — ты сам пол не понижаешь и не «оптимизируешь» неудобные пункты.
Метка `(срок: …)` — когда пункт должен быть удовлетворён: каркас → до фичи → предпрод.

## Ядро (применимо всегда)

- **AUTH-2 — Принцип наименьших привилегий (срок: каркас).** Deny-by-default. Каждая роль/токен — минимум прав. Роль admin существует всегда (см. AUTH-1).
- **SEC-1 — Безопасное хранение секретов (срок: каркас).** Никаких секретов в git и в собранных образах. Хранение по умолчанию — переменные GitLab CI; допустимы Vault / docker secrets.
- **DATA-1 — Классификация данных (срок: каркас).** До передачи любых данных в модель зафиксируй, что передавать допустимо. ПДн и коммерческую тайну без правовых оснований в модель не передавать.
- **DATA-2 — Политика по моделям и границе периметра (срок: каркас).** Зафиксируй разрешённые провайдеры/эндпоинты, границу периметра (внешний API vs on-prem) и какие данные покидают контур.
- **CICD-1 — Код хранится в GitLab (срок: каркас).** Репозиторий в GitLab, доступ по ролям.
- **CICD-3 — Ветвление, code review, защита основной ветки (срок: каркас).** Merge в основную ветку только через MR с обязательным ревью; прямой push в основную ветку запрещён.
- **DOC-1 — Описание проекта (срок: предпрод).** Краткое + полное описание: архитектура, схема потоков данных, какие данные обрабатываются (потоки данных — ссылкой на DATA-1/DATA-2, без дублирования).

## Для развёртываемого сервиса

_Если проект — не развёртываемый сервис (библиотека/скрипт/ноутбук), эти пункты снимает скилл
ce-brainstorm как N/A с записью причины. Сам не решай — оставляй как есть._

- **AUTH-1 — Bootstrap локального администратора (срок: каркас).** На первом запуске/деплое есть локальный администратор: предпочтительно через first-run в веб-панели, либо автосоздание при первом старте. Одного локального админа достаточно.
- **AUTH-3 — Безопасная локальная авторизация (срок: до фичи).** Политика паролей; защита от перебора; корректные сессии (HttpOnly+Secure cookie, таймаут неактивности, работающий выход).
- **ADM-1 — Административное управление только для админов (срок: до фичи).** Админ-функции только для роли admin. Ожидаемая форма — веб-админ-панель (в ней first-run и опциональная настройка LDAP). Для headless — задокументированный админ-интерфейс.
- **SEC-3 — Аудит-лог действий администратора (срок: до фичи).** Аудит-лог покрывает ВСЕ операции администратора. Человекочитаемый: кто → что сделал → над каким объектом. Объекты по наименованию (id — в скобках), не голым id.
- **DEPL-1 — Контейнеризация (срок: каркас).** По умолчанию Docker + docker compose; иные средства изоляции — по обоснованию.
- **DEPL-3 — Разделение окружений dev/test/prod (срок: каркас).** Три независимых окружения dev/test/prod с раздельными конфигами; секреты и адреса не пересекаются.
- **CICD-2 — Настроенный CI/CD (срок: каркас).** Пайплайн собирает артефакт и разворачивает его в целевую среду.
- **DOC-3 — Установка и развёртывание (срок: предпрод).** Установка, деплой и first-run (как создаётся первый локальный админ, см. AUTH-1).
- **DOC-4 — Руководство по использованию (срок: предпрод).** Как пользоваться сервисом с акцентом на UI; вставки про API где уместно; конкретные примеры.

## Чего НЕ делать

- Не коммить секреты и `.env` в git.
- Не добавляй домен/TLS/HTTPS-редиректы в дев-сборку: в разработке доступ по HTTP / IP / любому порту.
  Домен и TLS — позже на уровне прокси, это вне периметра проекта.
- Не понижай и не «упрощай» требования выше по своей инициативе. Сомнения по применимости — оставь
  скиллу ce-brainstorm.
- Не передавай в модель данные, не прошедшие классификацию (DATA-1).

## Что дальше

Полный опрос по специфике проекта, план разработки и его реализацию ведёт **скилл ce-brainstorm**.
Он читает `requirements-baseline.yaml`, разрешает условные пункты под этот проект и добавляет
проектные требования поверх этого пола.
